# Schema Reference — Stage 01 Schema Definition

This document describes the canonical schema generated by Stage 01:

```code
data/stage01_schema/schema.json
```

The schema is derived directly from the cleaned dataset produced by Stage 02:

```code
data/stage02_cleaned/cleaned_data.csv
```

All downstream stages (quality checks, reporting, orchestrator) depend on this schema.

---

## 1. Schema Structure

The schema is a JSON object with the following structure:

```json
{
  "fields": [
    {
      "name": "<column_name>",
      "type": "<data_type>",
      "description": "<human-readable description>",
      "required": <true|false>
    }
  ]
}
```

Each field corresponds to a column in the cleaned dataset.

---

## 2. Field Definitions

Below is the full reference for all fields in `schema.json`.

---

### Facility Information

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| facility_id | string | true | Unique facility identifier (POS/QIES) |
| facility_name | string | true | Official facility name |
| facility_type | string | true | Facility classification (SNF, HHA, Hospice, etc.) |
| address_line1 | string | true | Primary street address |
| address_line2 | string | false | Secondary address line |
| city | string | true | City name |
| state | string | true | Two‑letter state code |
| zip_code | string | true | 5‑digit ZIP code |
| county_name | string | false | County name |

---

### Provider & Certification

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| provider_number | string | true | CMS provider number (6 digits) |
| ccn | string | false | CMS Certification Number (alias of provider_number) |
| npi | string | false | National Provider Identifier (10 digits) |
| ownership_type | string | false | Ownership classification |
| medicare_certified | boolean | true | Medicare certification status |
| medicaid_certified | boolean | true | Medicaid certification status |

---

### Operational Status

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| operational_status | string | true | Active / Closed / Pending |
| status_effective_date | date | false | Date status became effective |
| last_updated | date | true | Last CMS update timestamp |

---

### POS‑Specific Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|

| pos_subregion | string | false | Sub‑region classification |

### QIES‑Specific Fields

### Geocoding & Location Normalization

| Field | Type | Required | Description |

|-------|------|----------|-------------|

| Field | Type | Required | Description |

---

## 3. Data Type Definitions

| Type | Description |
|------|-------------|
| string | UTF‑8 text, trimmed and normalized |
| boolean | true or false |
| float | Decimal number (no scientific notation) |
| date | ISO‑8601 (YYYY‑MM‑DD) |

---

## 4. Schema Evolution Rules

Stage 01 regenerates the schema automatically.  
To maintain stability:

### Adding a field

- Add to Stage 02 cleaned dataset  
- Add to schema.json  
- Update this document  
- Update Stage 03 quality checks  
- Update Stage 04 reporting if needed  

### Removing a field

- Remove from cleaned dataset  
- Remove from schema.json  
- Update this document  
- Validate downstream dependencies  

### Changing a field type

- Update Stage 02 cleaning logic  
- Update schema.json  
- Update Stage 03 quality checks
